# 上下文预算规则

保持协调器上下文精简的标准规则。在生成子 agent 或读取大量内容的工作流中引用此文档。

另请参阅：`references/universal-anti-patterns.md` 了解完整的通用规则集。

---

## 通用规则

每个生成 agent 或读取大量内容的工作流都必须遵循以下规则：

1. **绝不**读取 agent 定义文件（`agents/*.md`）—— `subagent_type` 会自动加载它们
2. **绝不**将大文件内联到子 agent 提示中 —— 告诉 agent 从磁盘读取文件
3. **读取深度随上下文窗口扩展** —— 检查 `.planning/config.json` 中的 `context_window_tokens`：
   - 小于 500000 个 token（默认 200k）：只读取 frontmatter、状态字段或摘要。绝不读取完整的 SUMMARY.md、VERIFICATION.md 或 RESEARCH.md 正文。
   - 大于等于 500000 个 token（1M 模型）：当内联展示或决策需要内容时，可以读取完整的子 agent 输出正文。仍然避免不必要的读取。
4. **委托**繁重工作给子 agent —— 协调器负责路由，不负责执行
5. **主动警告**：如果你已经消耗了大量上下文（大文件读取、多个子 agent 结果），请警告用户："上下文预算压力较大。请考虑检查点当前进度。"

## 按上下文窗口划分的读取深度

| 上下文窗口 | 子 Agent 输出读取 | SUMMARY.md | VERIFICATION.md | 其他阶段的 PLAN.md |
|---------------|------------------------|------------|-----------------|------------------------|
| < 500k（200k 模型） | 仅 Frontmatter | 仅 Frontmatter | 仅 Frontmatter | 仅当前阶段 |
| >= 500k（1M 模型） | 允许完整正文 | 允许完整正文 | 允许完整正文 | 仅当前阶段 |

**如何检查：** 读取 `.planning/config.json` 并检查 `context_window_tokens`。如果该字段不存在，按 200k 处理（保守默认值）。

## 上下文退化层级

监控上下文使用情况并相应调整行为：

| 层级 | 使用量 | 行为 |
|------|-------|----------|
| 峰值 | 0-30% | 完整操作。读取正文、生成多个 agent、内联结果。 |
| 良好 | 30-50% | 正常操作。优先 frontmatter 读取，积极委托。 |
| 退化 | 50-70% | 节约模式。仅读取 frontmatter，最少内联，向用户警告预算。 |
| 较差 | 70%+ | 紧急模式。立即检查点当前进度。除非关键，否则不进行新的读取。 |

## 上下文退化警告信号

质量会在触发恐慌阈值之前逐渐下降。注意这些早期信号：

- **静默的部分完成** —— agent 声称任务完成，但实现不完整。自检捕获文件是否存在，但不捕获语义完整性。始终验证 agent 输出是否满足计划的 must_haves，而不仅仅是文件存在。
- **逐渐模糊** —— agent 开始使用"适当处理"或"标准模式"等短语，而不是具体代码。这表明在预算警告触发之前就已存在上下文压力。
- **跳过步骤** —— agent 省略了通常会遵循的协议步骤。如果 agent 的成功标准有 8 项，但它只报告了 5 项，怀疑存在上下文压力。

在委托给 agent 时，协调器无法验证 agent 输出的语义正确性——只能验证结构完整性。这是一个根本性限制。通过 must_haves.truths 和抽检验证来缓解。
