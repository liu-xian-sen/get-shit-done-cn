# 思维模型：规划集群

供**规划器**和**路线图制作器** agent 使用的结构化推理模型。在计划创建过程中的决策节点应用这些模型，而非持续使用。每个模型都针对一个有据可查的失败模式。

来源：精选自 [thinking-partner](https://github.com/mattnowdev/thinking-partner) 模型目录（150+ 个模型）。选自与 GSD 规划工作流直接相关的模型。

## 冲突解决

前置验尸分析和约束分析都在不同粒度上分析风险。先运行约束分析（识别最难的约束），再运行前置验尸分析（围绕该约束及计划其余部分枚举失败模式）。

## 1. 前置验尸分析

**对抗：** 忽略失败模式的乐观计划分解。

在最终确定此计划之前，假设它已经失败了。列出失败的 3 个最可能原因——依赖缺失、分解错误、复杂度低估——并添加能尽早发现每种失败的缓解步骤或验收标准。

## 2. MECE 分解

**对抗：** 任务重叠（合并冲突）或任务缺口（需求缺失）。

在**需求**层级验证此任务分解是否 MECE：（1）列出阶段目标中的每个需求，（2）确认每个需求恰好映射到一个任务的 `<done>`，（3）如果两个任务修改同一文件，确认它们修改**不同**章节或服务**不同**需求，（4）标记任何未被任何任务覆盖的需求。

## 3. 约束分析

**对抗：** 将最难的约束推迟到最后一个任务，导致后期失败。

识别此阶段中的单一最难约束——如果它不起作用，其他一切都无关紧要。将该约束安排为任务 1 或 2，而不是最后。如果约束涉及外部 API 或不熟悉的库，在主要实现之前添加一个 spike/概念验证任务。

## 4. 可逆性测试

**对抗：** 对廉价决策过度分析，对代价高昂的决策分析不足。

对于此计划中的每个重要决策，将其分类为 REVERSIBLE（后期可以低成本更改）或 IRREVERSIBLE（后期更改需要迁移、破坏性更改或大量返工）。花费与不可逆性成比例的分析时间。对于不可逆决策，在计划中记录其原理。

## 5. 知识诅咒反制

**对抗：** 因指令压缩导致计划到执行器的歧义。

对于每个 `<action>` 步骤，像从未见过此代码库一样重新阅读。每个名词都明确吗（哪个文件？哪个函数？哪个端点？）？每个动词都具体吗（在哪里添加？如何修改？）？如果一个步骤可以有两种解释，重新措辞。在每个动作步骤中包含文件路径、函数名和预期行为。

## 6. 基础率忽视反制

**对抗：** 规划器忽视低置信度的研究警告。

在最终确定计划之前，读取 SUMMARY.md 中所有的 `[NEEDS DECISION]` 项目和 LOW 置信度建议。对于每个：要么（a）创建一个 `checkpoint:decision` 任务来解决它，要么（b）在计划的偏差注释中记录为何风险可以接受。被静默接受的 LOW 置信度项目会成为未文档化的技术债务。

## 差距闭合模式：根本原因检查

**仅在以下情况适用：** 规划器进入差距闭合模式（由 VERIFICATION.md 中的 `gaps_found` 触发）。

在编写修复计划之前，进行一轮"为什么"：为什么会出现这个差距？是计划缺陷（任务错误）、执行失误（正确的任务，错误的实现），还是假设改变（环境/依赖转变）？修复计划必须针对根本原因类别，而不仅仅是症状。

## 何时不需要思考

当情况不需要结构化推理模型时跳过：

- **单任务计划** —— 如果阶段有一个清晰的需求和一个明显的任务，不要运行前置验尸分析或 MECE 分析。直接写任务。
- **经过充分研究的阶段** —— 如果 RESEARCH.md 对每个决策都有 HIGH 置信度的建议且没有 `[NEEDS DECISION]` 项目，跳过基础率忽视反制。研究已经解决了不确定性。
- **修订迭代** —— 基于检查器反馈修订计划时，专注于修复被标记的问题。不要在每次修订时重新运行完整的模型套件——只应用与具体问题相关的模型（例如，如果检查器发现覆盖缺口，则使用 MECE）。
- **样板计划** —— 配置变更、版本升级、文档更新。这些没有值得进行前置验尸分析的失败模式。
