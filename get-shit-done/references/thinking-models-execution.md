# 思维模型：执行集群

供**执行器** agent 使用的结构化推理模型。在任务执行过程中的决策节点应用这些模型，而非持续使用。每个模型都针对一个有据可查的失败模式。

来源：精选自 [thinking-partner](https://github.com/mattnowdev/thinking-partner) 模型目录（150+ 个模型）。选自与 GSD 执行工作流直接相关的模型。

## 冲突解决

**强制函数和第一原理都推动"现在就做"。** 先运行第一原理（理解约束），再运行强制函数（创建机制）。是顺序的，而非竞争的。

## 1. 关注圈 vs 控制圈

**对抗：** 执行器试图修复其范围之外的事情——上游 bug、无关的技术债务、基础设施问题。

在修改计划 `<files>` 章节中未明确列出的任何代码之前，先问：这是在我的控制圈（计划范围）还是关注圈（我注意到但不应修复的事情）？如果是关注圈：将其记录为偏差注释或已推迟项目，**不要**修复它。执行器的工作是构建计划规定的内容，而不是改进代码库。"顺便修一下"引起的范围蔓延是执行器超支的第一大原因。

## 2. 强制函数

**对抗：** 将困难决策推迟到运行时而不是在构建时解决。

当你遇到模糊的需求或不清晰的集成点时，创建一个强制函数，使决策**现在**就变得明确，而不是把它藏在 TODO 或运行时检查后面。示例：使用 TypeScript `never` 类型强制穷举 switch，为必需的配置值添加构建时断言，创建强制调用者处理错误情况的接口。如果决策确实无法在构建时做出，将其记录为 `checkpoint:decision` 偏差——不要静默推迟。

## 3. 第一原理思维

**对抗：** 从其他代码复制模式而不理解它们是否适合当前任务。

在从另一个文件或阶段复制模式之前，分解该模式存在的**原因**：它满足什么约束？你当前的任务有相同的约束吗？如果没有，该模式可能是照猫画虎。从任务的实际需求构建你的实现，而不是从最近的现有示例。如有疑问，计划的 `<action>` 步骤定义了要构建什么——从这些步骤推导实现，而不是从相邻代码。

## 4. 奥卡姆剃刀

**对抗：** 用不必要的抽象、泛型或未来设计来过度设计简单任务。

在添加抽象层、泛型类型参数、工厂模式或配置选项之前，问：计划**需要**这种灵活性吗？如果计划说"创建一个执行 X 的函数"，就创建一个执行 X 的函数——而不是一个理论上可以通过 Y 通过 Z 来执行 X 的可配置、可扩展、可插拔的框架。满足计划 `<done>` 条件的最简单实现就是正确的。只有当计划明确要求时才增加复杂性。

## 5. 切斯特顿的围栏

**对抗：** 不理解现有代码为何如此编写就删除或修改它。

在删除、替换或大幅修改计划涉及的现有代码之前，确定它为何存在。检查：git blame 引入它的提交、解释原理的注释、测试它的测试用例、创建它的 PLAN.md 或 SUMMARY.md。如果目的不清楚，保留它并添加注释指出不确定性——**不要**删除你不理解其目的的代码。如果计划明确说要删除它，仍然在偏差注释中记录它做了什么。

---

## 何时不需要思考

当情况不需要结构化推理模型时跳过：

- **直截了当的任务动作** —— 如果计划说"用内容 Y 创建文件 X"且动作不含糊，直接执行。不要调用第一原理来分析为什么要创建计划让你创建的文件。
- **遵循已建立的项目模式** —— 如果代码库有清晰、一致的模式（例如每个路由处理程序遵循相同结构），且计划说要添加另一个，就遵循该模式。切斯特顿的围栏适用于删除模式，而非遵循模式。
- **琐碎的文件编辑** —— 添加导入、修复拼写错误、更新版本号。这些是不涉及设计决策的机械性更改。
- **运行验证命令** —— 执行计划的 `<verify>` 步骤是程序性的。只有当验证步骤失败且需要决定如何响应时，才调用模型。
