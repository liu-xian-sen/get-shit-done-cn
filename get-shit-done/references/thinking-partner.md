# Thinking Partner 集成

在工作流决策节点进行有条件的扩展思考。当 `.planning/config.json` 中 `features.thinking_partner: true` 时激活（默认：false）。

---

## 权衡检测信号

当开发者响应中包含表示竞争优先级的特定信号时，thinking partner 被激活：

**关键词信号：**
- "或" / "versus" / "vs" 连接两种方法
- "权衡" / "trade-off" / "tradeoffs"
- "一方面" / "另一方面"
- "利弊"
- "不确定选哪个" / "纠结于"

**结构性信号：**
- 开发者列出 2+ 个竞争选项
- 开发者问"哪个更好"或"你推荐什么"
- 开发者撤回之前的决定（"其实，也许我们应该..."）

**何时不激活：**
- 开发者已经做出明确选择
- "或"是修辞性的或微不足道的（例如，"制表符还是空格"——使用项目约定）
- 简单的是/否问题
- 开发者明确要求继续前进

---

## 集成点

### 1. 讨论阶段 —— 权衡深入分析

**时机：** 在 `discuss_areas` 步骤期间，开发者的回答揭示了竞争优先级。

**做什么：** 暂停正常的问题流程，提供简短的结构化分析：
```
我注意到这里存在竞争优先级——{X} 为 {A} 优化，而 {Y} 为 {B} 优化。

需要我在我们决定之前分析这些权衡吗？
[是的，分析权衡] / [不，我已经决定了]
```

如果是，提供简短的（3-5 条）分析，覆盖：
- 每种方法优化的内容
- 每种方法牺牲的内容
- 哪种更符合项目的既定目标（来自 PROJECT.md）
- 有推理的建议

然后返回正常的讨论流程。

### 2. 规划阶段 —— 架构决策分析

**时机：** 在步骤 11（处理检查器返回）期间，当 plan-checker 标记包含架构权衡关键词的问题时。

**做什么：** 在发送到修订循环之前，分析架构决策：
```
plan-checker 标记了一个架构权衡：{问题描述}

简短分析：
- 选项 A：{方法} —— {优/缺点}
- 选项 B：{方法} —— {优/缺点}
- 建议：{选择}，因为 {与阶段目标一致的推理}

将此建议应用于修订吗？[是] / [不，由我决定]
```

### 3. 探索 —— 方法比较（需要 #1729）

**时机：** 在苏格拉底对话期间，当出现多种可行方法时。
**注意：** 当 /gsd-explore（#1729）落地时，将添加此集成点。

---

## 配置

```json
{
  "features": {
    "thinking_partner": true
  }
}
```

默认：`false`。Thinking partner 是选择性加入的，因为它会增加交互式工作流的延迟。

---

## 设计原则

1. **轻量级** —— 内联分析，而非独立的交互式会话
2. **选择性加入** —— 必须明确启用，绝不默认激活
3. **可跳过** —— 始终提供"不，我已经决定了"来绕过
4. **简短** —— 最多 3-5 条，而不是完整的研究报告
5. **对齐** —— 建议在可用时引用 PROJECT.md 目标
