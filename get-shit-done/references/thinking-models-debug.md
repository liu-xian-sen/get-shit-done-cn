# 思维模型：调试集群

供**调试器** agent 使用的结构化推理模型。在调查过程中的决策节点应用这些模型，而非持续使用。每个模型都针对一个有据可查的失败模式。

来源：精选自 [thinking-partner](https://github.com/mattnowdev/thinking-partner) 模型目录（150+ 个模型）。选自与 GSD 调试工作流直接相关的模型。

## 冲突解决

**故障树分析和假设驱动调查是顺序的：** 先进行故障树分析（生成可能原因树），再进行假设驱动调查（系统地测试每个分支）。故障树提供地图；假设驱动提供遍历的纪律性。

## 1. 故障树分析

**对抗：** 在未系统地映射失败路径之前就跳到结论。

在测试任何假设之前，构建故障树：以观察到的症状作为根节点，然后在每一层分支出所有可能的原因（硬件、软件、配置、数据、环境）。使用 AND/OR 门——某些失败需要多个条件（AND），其他有独立触发器（OR）。这棵树成为你的调查路线图。按可能性和可测试性对分支排列优先级，但**不要**因为看起来不太可能就剪枝——容易测试的低可能性原因应该提前测试。

## 2. 假设驱动调查

**对抗：** 随机修改并希望某些改变有效——"霰弹枪调试"反模式。

对于故障树中的每个假设，遵循严格协议：PREDICT（"如果假设 H 正确，那么测试 T 应该产生结果 R"），TEST（执行恰好一个测试），OBSERVE（记录实际结果），CONCLUDE（匹配 = 支持，不匹配 = 排除，意外 = 新证据）。绝不跳过 PREDICT 步骤——没有预测，就无法区分有意义的结果和噪声。每次测试绝不改变超过一个变量——如果你改变了两件事后 bug 消失了，你不知道哪个修改起了作用。

## 3. 奥卡姆剃刀

**对抗：** 在简单解释尚未排除时追求复杂的解释。

在调查复杂的多组件交互 bug、竞态条件或框架级问题之前，先验证简单解释：变量名拼写错误、文件路径错误、缺少导入、配置值不正确、缓存过时、环境变量错误。这些"无聊"的原因占大多数 bug。只有在简单原因**排除之后**才升级到复杂假设。如果你当前的假设需要 3+ 件事同时出错，退一步寻找单点失败。

## 4. 反事实思维

**对抗：** 不问"如果只改变这一件事会怎样？"而无法隔离因果关系。

当你对根本原因有假设时，构建反事实："如果我**只**改变这一个变量/配置/行，bug 应该消失（或出现）。"执行反事实测试。如果在你的针对性修改后 bug 仍然存在，你的假设是错误的——原因在别处。如果 bug 消失了，你有了强有力的因果证据。这比相关性（"bug 在部署 X 后出现"）更有力，因为它测试的是机制，而非时间线。

## 何时不需要思考

当情况不需要结构化推理模型时跳过：

- **明显的单一原因 bug** —— 如果错误消息指出了确切的文件、行和原因（例如 `TypeError: Cannot read property 'x' of undefined at foo.js:42`），直接修复。不要为带有堆栈跟踪的空引用构建故障树。
- **重现已知修复** —— 如果你已经从之前的调查中知道根本原因，或者用户告诉你确切出了什么问题，跳过假设驱动调查，直接进行修复。
- **拼写错误、缺少导入、路径错误** —— 如果奥卡姆剃刀能立即解决，直接应用修复，无需调用完整模型。该模型存在于简单检查失败时，而不是作为简单检查的门控。
- **读取错误日志** —— 读取和理解错误输出是正常的调试，不是"决策节点"。只有当你有多个合理的假设且需要决定先测试哪个时，才调用模型。
