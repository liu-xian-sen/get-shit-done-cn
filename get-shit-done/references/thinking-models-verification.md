# 思维模型：验证集群

供**验证器**和 **plan-checker** agent 使用的结构化推理模型。在验证过程中应用这些模型，而非持续使用。每个模型都针对一个有据可查的失败模式。

来源：精选自 [thinking-partner](https://github.com/mattnowdev/thinking-partner) 模型目录（150+ 个模型）。选自与 GSD 验证工作流直接相关的模型。

## 冲突解决

**反转**和**确认偏见反制**都在寻找失败，但目的不同。按顺序运行：

1. **先运行反转**（头脑风暴）：生成 3 种可能出错的方式
2. **再运行确认偏见反制**（结构化检查）：找到一个部分满足的需求、一个误导性测试、一个未覆盖的错误路径

反转生成列表；确认偏见反制是验证其中项目的纪律性。

## 1. 反转

**对抗：** 验证器确认成功而不是发现失败。

不要检查什么是**正确的**，而是列出此实现尽管通过测试但可能**错误**的 3 种具体方式：缺失的边界情况、静默的数据丢失、竞态条件、未处理的错误路径。对于每一种，写出具体检查（grep 模式、用特定输入测试、验证错误处理存在）。此外，检查 SUMMARY.md 中任何记录的**偏差**是否改变了 must-have 的含义或适用性。如果 must-have 是基于方法 A 编写的，但执行器使用了方法 B，那么 must-have 可能需要重新解释，而不是字面检查。

## 2. 切斯特顿的围栏

**对抗：** 将有目的的代码标记为无用或不必要。

在将任何现有代码标记为无用、冗余或过度复杂之前，确定它为何如此编写。检查 git blame、注释、测试用例和创建它的 PLAN.md。如果原因不清楚，标记为"目的不明——建议保留并附 WARNING，不删除"，并包含引入该提交的 git blame 哈希。

## 3. 确认偏见反制

**对抗：** 受 SUMMARY.md 声明影响的验证器预先看到成功。

在初始验证过后，进行**反确认**过：（1）找到一个只部分满足的需求，（2）找到一个通过但实际上没有测试所述行为的测试，（3）找到一个没有测试覆盖的错误路径。即使整体验证通过，也要报告这些。

## 4. 规划谬误校准

**对抗：** 将范围过大的计划当作合理的接受（plan-checker）。

对于每个被估计为"简单"或"小"的任务，检查：它触及超过 2 个文件吗？它需要理解一个不熟悉的 API 吗？它修改共享基础设施吗？如果任意一项为是，标记为可能被低估。任务超过 5 个或每个任务触及超过 4 个文件的计划是范围过大的。

## 5. 反事实思维

**对抗：** 在每个步骤都假设成功且没有错误恢复的计划（plan-checker）。

对于每个计划，问："如果执行器**完全按照**此计划操作，但遇到常见失败会怎样：依赖版本不匹配、API 返回意外格式、文件已被前一个计划修改？"如果计划没有应急路径且 `<action>` 步骤在每一点都假设成功，标记为 WARNING："任务 T{n} 没有错误恢复路径。"

---

## 何时不需要思考

当情况不需要结构化推理模型时跳过：

- **重新验证之前通过的项目** —— 在重新验证模式下，之前通过初始检查的项目只需要快速回归检查（存在性 + 基本完整性检查），而不是完整的反转 + 确认偏见反制处理。
- **二元存在性检查** —— 如果 must-have 是"文件 X 存在且 >N 行"且文件明显存在且有实质内容，不要对其运行反事实思维。将模型保留给模糊或依赖接线的 must-have。
- **直接的测试结果** —— 如果 `<verify>` 命令产生清晰的通过/失败输出（例如测试套件以 0 退出且所有测试通过），接受结果。只有在测试结果模糊或你怀疑测试实际上没有测试它们声称要测试的内容时才调用模型。
- **INFO 级别问题** —— 不要应用结构化推理来决定一个 INFO 级别的观察是否实际上是 BLOCKER。INFO 项目按定义是信息性的，永远不触发门禁。
