# AI 评估参考

> 供 `gsd-eval-planner` 和 `gsd-eval-auditor` 使用的参考文档。
> 基于"AI Evals for Everyone"课程（Reganti & Badam）及行业实践。

---

## 核心概念

### 评估存在的原因
AI 系统具有非确定性。在不同运行、用户或边界情况下，输入 X 不能可靠地产生输出 Y。评估是持续评估系统行为是否在真实世界条件下满足预期的过程——仅靠单元测试和集成测试是不够的。

### 模型评估与产品评估
- **模型评估**（MMLU、HumanEval、GSM8K）—— 在标准化条件下衡量通用能力。仅用作初始筛选。
- **产品评估** —— 在你的具体系统中、用你的数据、你的用户、你的领域规则来衡量行为。这是 80% 评估精力应投入的地方。

### 每个评估的三个组成部分
- **输入** —— 影响系统的一切：查询、历史记录、检索的文档、系统提示、配置
- **预期** —— 良好行为的样子，通过评分标准定义
- **实际** —— 系统产生的内容，包括中间步骤、工具调用和推理轨迹

### 三种测量方法
1. **基于代码的指标** —— 确定性检查：JSON 验证、必要的免责声明、性能阈值、分类标志。快速、廉价、可靠。优先使用。
2. **LLM 评审** —— 一个模型根据评分标准评估另一个模型。对于主观质量（语气、推理、升级）很有效。在信任之前需要与人工判断进行校准。
3. **人工评估** —— 细致判断的黄金标准。无法扩展。用于校准、边界情况、定期抽样和高风险决策。

最有效的系统结合使用三种方法。

## 评估维度

### 部署前（开发阶段）

| 维度 | 测量内容 | 适用场景 |
|-----------|-----------------|-----------------|
| **事实准确性** | 对照基准事实的声明正确性 | RAG、知识库、任何事实性断言 |
| **上下文忠实度** | 回应是否基于提供的上下文而非凭空捏造 | RAG 流水线、文档问答、检索增强系统 |
| **幻觉检测** | 貌似合理但无依据的声明 | 所有生成式系统、高风险领域 |
| **升级准确性** | 正确识别何时需要人工干预 | 客服、医疗、金融咨询 |
| **政策合规性** | 遵守业务规则、法律要求、免责声明 | 受监管行业、企业部署 |
| **语气/风格适当性** | 与品牌声音、受众预期、情感背景的匹配 | 面向客户的系统、内容生成 |
| **输出结构有效性** | schema 合规性、必填字段、格式正确性 | 结构化提取、API 集成、数据管道 |
| **任务完成度** | 系统是否完成了既定目标 | 智能体工作流、多步骤任务 |
| **工具使用正确性** | 工具的正确选择和调用 | 有工具调用的 agent 系统 |
| **安全性** | 不存在有害、偏颇或不当的输出 | 所有面向用户的系统 |

### 生产监控

| 维度 | 监控方法 |
|-----------|---------------------|
| **安全违规** | 在线护栏 —— 实时，立即干预 |
| **合规失败** | 在线护栏 —— 在用户看到输出前阻止或升级 |
| **质量下降趋势** | 离线飞轮 —— 对抽样交互进行批量分析 |
| **新出现的失败模式** | 信号-指标偏差 —— 当用户行为信号与指标得分出现偏差时，手动调查 |
| **成本/延迟漂移** | 基于代码的指标 —— 自动阈值告警 |

---

## 护栏与飞轮的决策

问：「如果这种行为出错，对我的业务是否会是灾难性的？」

- **是 → 护栏** —— 在线运行、实时、立即干预（阻止、升级、移交）。要有选择性：护栏会增加延迟。
- **否 → 飞轮** —— 作为批量分析离线运行，随时间推动系统改进。

## 评分标准设计

没有上下文的通用指标毫无意义。"有用性"在房地产中意味着清晰地总结房源。在医疗保健中，它意味着知道何时*不*回答。

评分标准必须定义：
1. 被衡量的维度
2. 5 分制中 1、3、5 分的得分内容（或通过/失败标准）
3. 针对特定领域的可接受与不可接受行为示例

没有评分标准，LLM 评审产生的是噪声而非信号。

## 参考数据集指南

- 从 **10-20 个高质量示例**开始——而不是 200 个平庸的示例
- 覆盖：关键成功场景、常见用户工作流、已知边界情况、历史失败模式
- 让领域专家标注示例（而不只是工程师）
- 根据生产中学到的内容扩展——不要为假设的覆盖范围而构建

## 评估工具指南

| 工具 | 类型 | 最适用 | 核心优势 |
|------|------|----------|-------------|
| **RAGAS** | Python 库 | RAG 评估 | 专为此目的设计的指标：忠实度、答案相关性、上下文精确度/召回率 |
| **Langfuse** | 平台（开源、可自托管） | 所有系统类型 | 强大的追踪、提示管理，适合需要基础设施控制的团队 |
| **LangSmith** | 平台（商业） | LangChain/LangGraph 生态系统 | 与 LangChain 集成最紧密；已在该生态中时是最佳选择 |
| **Arize Phoenix** | 平台（开源 + 托管） | RAG + 多 agent 追踪 | 强大的 RAG 评估 + 追踪可视化；开源并有托管选项 |
| **Braintrust** | 平台（商业） | 模型无关评估 | 数据集和实验管理；适合跨框架比较 |
| **Promptfoo** | CLI 工具（开源） | 提示测试、CI/CD | CLI 优先，非常适合 CI/CD 提示回归测试 |

### 按系统类型选择工具

| 系统类型 | 推荐工具 |
|-------------|---------------------|
| RAG / 知识问答 | RAGAS + Arize Phoenix 或 Braintrust |
| 多 agent 系统 | Langfuse + Arize Phoenix |
| 对话式 / 单模型 | Promptfoo + Braintrust |
| 结构化提取 | Promptfoo + 基于代码的验证器 |
| LangChain/LangGraph 项目 | LangSmith（原生集成） |
| 生产监控（所有类型） | Langfuse、Arize Phoenix 或 LangSmith |

## 开发生命周期中的评估

### 计划阶段（评估感知设计）
在编写代码之前，定义：
1. 正在构建的 AI 系统类型 → 决定框架和主要评估关注点
2. 关键失败模式（3-5 个不能出错的行为）
3. 评分标准 —— 每个维度可接受/不可接受行为的明确定义
4. 评估策略 —— 哪些维度使用代码指标、LLM 评审或人工审查
5. 参考数据集要求 —— 大小、构成、标注方法
6. 评估工具选择

输出：AI-SPEC.md 的 EVALS-SPEC 章节

### 执行阶段（边构建边植入）
- 从第一天起就添加追踪（Langfuse、Arize Phoenix 或 LangSmith）
- 与实现并行构建参考数据集
- 先实现基于代码的检查；仅对主观维度添加 LLM 评审
- 通过 Promptfoo 或 Braintrust 在 CI/CD 中运行评估

### 验证阶段（部署前验证）
- 对所有指标运行完整参考数据集
- 对边界情况和 LLM 评审分歧进行人工审查
- 对照人工分数校准 LLM 评审（目标相关性 ≥ 0.7 再信任）
- 定义并配置生产护栏
- 建立监控基准

### 监控阶段（生产评估循环）
- 智能抽样 —— 对有问题信号的交互加权（重试、异常长度、明确升级）
- 每次交互运行在线护栏
- 对抽样批次运行离线飞轮
- 关注信号-指标偏差 —— 评估差距的早期预警系统

## 常见陷阱

1. **假设基准测试能预测产品成功** —— 它们不能；模型评估是筛选器，不是判决
2. **孤立地设计评估** —— 领域专家必须共同定义评分标准；仅靠工程师会遗漏关键细节
3. **第一天就构建全面覆盖** —— 从小处开始（10-20 个示例），从真实失败模式扩展
4. **信任未校准的 LLM 评审** —— 在依赖前，先对照人工判断进行验证
5. **测量一切** —— 只追踪驱动决策的指标；"全部收集"会产生噪声
6. **将评估视为一次性设置** —— 用户行为在演变，需求在变化，失败模式在浮现；评估是持续性的
