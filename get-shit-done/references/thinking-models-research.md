# 思维模型：研究集群

供**研究员**和**综合器** agent 使用的结构化推理模型。在研究和综合过程中的决策节点应用这些模型，而非持续使用。每个模型都针对一个有据可查的失败模式。

来源：精选自 [thinking-partner](https://github.com/mattnowdev/thinking-partner) 模型目录（150+ 个模型）。选自与 GSD 研究工作流直接相关的模型。

## 冲突解决

**第一原理思维和钢铁人都在扩展范围**——先运行第一原理（分解问题），再运行钢铁人（强化备选方案）。不要同时运行。

## 1. 第一原理思维

**对抗：** 不分解到基本组件就接受表面解释。

在接受任何技术建议或架构模式之前，将其分解到基本约束：这解决什么问题？不可谈判的需求是什么？有哪些物理/逻辑限制？从这些约束**自下而上**构建你的建议，而不是从传统智慧**自上而下**。如果你无法从第一原理解释为什么建议是正确的，无论来源数量多少，都将其标记为 `[LOW]`。

## 2. 辛普森悖论意识

**对抗：** 综合器在不检查混淆分裂的情况下聚合相互冲突的研究。

在合并显示矛盾结果的多个研究文档的发现时，检查当你按隐藏变量分裂时矛盾是否消失：框架版本、部署目标、项目规模或用例类别。整体上基准测试更快的库对**你的**特定工作负载可能更慢。在通过多数投票解决矛盾之前，问："是否存在能解释两种发现在各自背景下都正确的子群体分裂？"

## 3. 幸存者偏见

**对抗：** 只找到成功案例而遗漏失败和被放弃的方法。

在收集支持推荐方法的证据后，主动搜索**放弃**它的项目。检查 GitHub issues 中的"migrated away from"、"replaced X with"或"problems with X at scale"。一个有 10 个成功故事和 100 个悄然失败的技术看起来很棒，直到你检查墓地。对负面证据（迁移-离开故事、弃用通知、未解决的问题）**比**正面证据赋予**更高**权重——失败是被低估报告的。

## 4. 确认偏见反制

**对抗：** 搜索确认初始假设的证据，同时忽略不确认的证据。

在形成初始建议后，花一个完整的研究周期**反对**它。使用搜索词如"{技术} problems"、"{技术} alternatives"、"why not {技术}"、"{技术} vs {竞争者}"。对于找到的每个不确认证据，要么（a）用更高置信度的来源反驳它，要么（b）将其作为建议的警告添加。如果你找不到**任何**对建议的批评，你的搜索范围太窄了——扩大它。

## 5. 钢铁人

**对抗：** 不给替代方法最强可能的形式就否定它。

在建议反对某个替代技术或方法之前，构建其**最强**可能的论据。一个充满热情的倡导者会说什么？它在哪些用例中比你的建议更好？哪些权衡有利于它？将钢铁人化的备选方案与你的建议并排展示，并进行诚实的比较。如果钢铁人化的备选方案具有竞争力，将决策标记为 `[NEEDS DECISION]` 而不是单方面建议。

---

## 何时不需要思考

当情况不需要结构化推理模型时跳过：

- **来自 CONTEXT.md 的已锁定决策** —— 如果用户已经决定"使用库 X"，不要对备选方案运行钢铁人分析或对选择进行第一原理分解。研究如何好好使用 X，而不是 X 是否是正确的选择。
- **标准技术栈查找** —— 如果你只是在检查知名库的最新版本或阅读其 API 文档，不要调用幸存者偏见或确认偏见反制。这些模型用于评估有争议的建议，而非事实查找。
- **单一技术阶段** —— 如果阶段涉及一个没有备选方案需要评估的技术（例如，"添加 ESLint 规则 X"），跳过比较模型（钢铁人、确认偏见反制）。只需研究实现。
- **仅限代码库的研究** —— 如果研究是纯内部的（理解现有代码模式、找到函数被调用的位置），结构化推理模型没有价值。使用 grep 并读代码。
