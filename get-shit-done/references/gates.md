# 门禁分类

GSD 工作流中使用的规范门禁类型。每个验证检查点都对应以下四种类型之一。

---

## 门禁类型

### 预检门禁
**目的：** 在开始操作之前验证前置条件。
**行为：** 条件未满足时阻止进入。不创建任何部分工作。
**恢复：** 修复缺少的前置条件，然后重试。
**示例：**
- plan-phase 在规划前检查 REQUIREMENTS.md 是否存在
- execute-phase 在执行前验证 PLAN.md 是否存在
- discuss-phase 确认阶段存在于 ROADMAP.md 中

### 修订门禁
**目的：** 评估输出质量，如果不足则路由到修订。
**行为：** 携带具体反馈回溯到生产者。受迭代上限约束。
**恢复：** 生产者处理反馈；检查器重新评估。如果连续迭代之间问题数量没有减少（停滞检测），循环也会提前升级。达到最大迭代次数后无条件升级。
**示例：**
- plan-checker 审查 PLAN.md（最多 3 次迭代）
- Verifier 对照成功标准检查阶段交付物

### 升级门禁
**目的：** 将无法解决的问题上报给开发者做决策。
**行为：** 暂停工作流，展示选项，等待人工输入。
**恢复：** 开发者选择操作；工作流在选定路径上恢复。
**示例：**
- 3 次迭代后修订循环耗尽
- worktree 清理期间的合并冲突
- 需要澄清的模糊需求

### 终止门禁
**目的：** 终止操作以防止损害或浪费。
**行为：** 立即停止，保存状态，报告原因。
**恢复：** 开发者调查根本原因，修复后从检查点重启。
**示例：**
- 执行期间上下文窗口严重不足
- STATE.md 处于错误状态，阻止 /gsd-next
- 验证发现关键交付物缺失

## 门禁矩阵

| 工作流 | 阶段 | 门禁类型 | 检查的制品 | 失败行为 |
|----------|-------|-----------|-------------------|------------------|
| plan-phase | 入口 | 预检 | REQUIREMENTS.md、ROADMAP.md | 阻止并显示文件缺失消息 |
| plan-phase | 步骤 12 | 修订 | PLAN.md 质量 | 回溯到 planner（最多 3 次） |
| plan-phase | 修订后 | 升级 | 未解决的问题 | 上报给开发者 |
| execute-phase | 入口 | 预检 | PLAN.md | 阻止并显示计划缺失消息 |
| execute-phase | 完成 | 修订 | SUMMARY.md 完整性 | 重新运行未完成的任务 |
| verify-work | 入口 | 预检 | SUMMARY.md | 阻止并显示摘要缺失 |
| verify-work | 评估 | 升级 | 失败的标准 | 向开发者展示差距 |
| next | 入口 | 终止 | 错误状态、检查点 | 停止并显示诊断信息 |

## 实现门禁

在设计或审计工作流验证点时使用此分类：

- **预检**门禁属于工作流入口点。它们是廉价的、确定性的检查，可以防止浪费工作。如果可以用文件存在性检查或配置读取来验证前置条件，使用预检门禁。
- **修订**门禁属于质量可变的生产者步骤之后。始终将其与迭代上限配对以防止无限循环。上限应反映每次迭代的成本——代价高的操作应减少重试次数。
- **升级**门禁属于自动解决不可能或有歧义的地方。它们是修订循环和终止之间的安全阀。向开发者展示清晰的选项和足够的上下文以供决策。
- **终止**门禁属于继续会造成损害、浪费大量资源或产生无意义输出的节点。它们应保存状态，以便在根本原因修复后恢复工作。

**选择启发：** 从预检开始。如果检查发生在工作产出之后，是修订门禁。如果修订循环无法解决问题，升级。如果继续有危险，终止。
