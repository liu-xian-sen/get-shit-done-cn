<purpose>
在执行前分析 ROADMAP.md 各阶段之间的依赖关系。检测阶段间的文件重叠、语义 API/数据流依赖，并建议添加 `Depends on` 条目，以防止 `/gsd-manager` 并行执行时产生合并冲突。
</purpose>

<process>

## 1. 加载 ROADMAP.md

读取 `.planning/ROADMAP.md`。若不存在，报错："未找到 ROADMAP.md — 请先运行 `/gsd-new-project`。"

提取所有阶段。对每个阶段记录：
- 阶段编号和名称
- 范围/目标描述
- `Files` 或 `files_modified` 字段中列出的文件（如有）
- 现有 `Depends on` 字段值

## 2. 推断可能修改的文件

对于没有明确 `files_modified` 的每个阶段，分析范围/目标描述以推断哪些文件可能会被修改。使用以下启发式规则：

- **数据库/模式阶段** → 迁移文件、模式定义、模型文件
- **API/后端阶段** → 路由文件、控制器文件、服务文件、处理器文件
- **前端/UI 阶段** → 组件文件、页面文件、样式文件
- **认证阶段** → 中间件文件、认证路由文件、会话/令牌文件
- **配置/基础设施阶段** → 配置文件、环境文件、CI/CD 文件
- **测试阶段** → 测试文件、规范文件、固件文件
- **共享工具阶段** → lib/utils 文件、共享类型定义

按推断的文件域（数据库、API、前端、认证、配置、共享）对阶段分组。

## 3. 检测依赖关系

对每对阶段 (A, B)，检查依赖信号：

### 文件重叠检测
若阶段 A 和 B 都会修改同一域或同一具体文件，则其中一个必须在另一个之前运行。*提供基础* 的阶段先运行。

### 语义依赖检测
读取每个阶段的范围/目标，查找以下模式：
- 阶段 B 提到消费、使用或调用阶段 A 创建/实现的内容
- 阶段 B 引用阶段 A 构建的"API"、"模式"、"模型"、"端点"或"接口"
- 阶段 B 说"在 X 完成后"、"一旦 X 构建好"、"使用第 N 阶段的 X"
- 阶段 B 扩展或修改阶段 A 建立的代码

### 数据流检测
- 阶段 A 创建数据结构、模式或类型 → 阶段 B 消费或转换它们
- 阶段 A 迁移数据库 → 阶段 B 从该数据库读取
- 阶段 A 暴露 API 契约 → 阶段 B 实现该契约的客户端

## 4. 构建依赖表

输出依赖建议表：

```
阶段依赖分析
=========================

第 N 阶段：<名称>
  范围：<简要范围>
  可能涉及：<推断的文件域>

  建议的依赖：
  → Depends on: 第 M 阶段 — 原因：<重叠/语义/数据流说明>

  当前 "Depends on"：<现有值或"（无）">
```

对于未检测到依赖的阶段对，说明："未检测到第 X 阶段与第 Y 阶段之间的依赖。"

## 5. 汇总建议修改

展示 ROADMAP.md `Depends on` 建议修改的整合差异：

```
ROADMAP.md 建议更新：
  第 3 阶段：添加 "Depends on: 1, 2"   （文件重叠：数据库模式）
  第 5 阶段：添加 "Depends on: 3"      （语义：使用第 3 阶段的认证 API）
  第 4 阶段：无需修改                  （独立范围）
```

## 6. 确认并应用

询问用户："是否将这些 `Depends on` 建议应用到 ROADMAP.md？（yes / no / edit）"

- **yes** — 将所有建议的 `Depends on` 条目写入 ROADMAP.md。确认每次写入。
- **no** — 仅以文本形式打印建议。用户手动更新。
- **edit** — 逐条呈现，每条建议单独询问 yes/no/skip。

写入 ROADMAP.md 时：
- 找到阶段条目并添加或更新 `Depends on:` 字段
- 保留所有其他阶段内容不变
- 不重新排序阶段

应用后显示："ROADMAP.md 已更新。运行 `/gsd-manager` 以按正确顺序执行各阶段。"

</process>
